## 問題① 総借入/売上の極端な外れ値 → 【誤検知・撤回】

### 結論

当初 `debt_to_sales = (bank_credit + lease_credit) / nenshu` の比率が極端（最大 80,050%）な 101 件を単位不整合（`unit_mismatch`）と疑ったが、**これはデータ品質問題ではなかった**。

- `bank_credit` は対象ユーザー企業の**銀行融資残高**を表すフィールドであり、**売上（nenshu）に対して大きくなるのは正常**な経済実態（借入の方が年商より大きい企業は普通に存在する）。
- 「総借入が年商の3倍を超えるのは不自然」という当初の前提が誤りだった。比率の高さ自体は単位ズレの証拠にならない。
- よって `unit_mismatch` フラグ（101件、内 `lost_reason_missing` 併記 11件）は **2026-06-03 に全件撤回**した。

```sql
-- 撤回に使用した UPDATE
UPDATE past_cases SET quality_flag =
  CASE
    WHEN quality_flag = 'unit_mismatch' THEN NULL
    WHEN quality_flag = 'lost_reason_missing,unit_mismatch' THEN 'lost_reason_missing'
    ELSE quality_flag
  END;
```

### 教訓

`bank_credit` / `lease_credit` は当社与信ではなく**対象ユーザーの総与信（銀行融資・リース残高）**。比率ベースの外れ値検出は、フィールドの意味を取り違えると誤検知を生む。今後この比率を品質シグナルとして使わない。

## 問題② 失注理由の未入力

### 検出ロジック

`lost_reason` は **カラムではなく** JSON `data.lost_reason` に格納。空文字ではなく **プレースホルダ文字列「理由未入力」** で未入力が表現されていた。

- 検出条件: `final_status='失注'` かつ `data.lost_reason ∈ {"", "理由未入力"}` → `lost_reason_missing`
- 該当: **317 / 750 件（42.3%）**
- Q_risk が報告した「49%」は **高スコア(≥60)失注 293件中 143件（49%）** が未入力、を指す（再現確認済み）。

### 入力済み lost_reason 内訳（参考）

| 理由 | 件数 |
|------|-----:|
| 理由未入力（=未入力） | 317 |
| 他社競合（レート） | 201 |
| 調達方法変更 | 118 |
| 設備見合わせ | 56 |
| その他（不成約） | 26 |
| 他社競合（その他） | 21 |
| 物件不適 | 7 |
| 業績不振 | 4 |

### 営業部別の偏り（最重要）

| 営業部 | 未入力 | 失注総数 | 未入力率 |
|--------|------:|--------:|--------:|
| **埼玉営業部** | **235** | **235** | **100%** |
| 宇都宮営業部 | 42 | 153 | 27% |
| 小山営業部 | 12 | 86 | 14% |
| 足利営業部 | 27 | 265 | 10% |

→ **埼玉営業部は失注の全件が理由未入力**。同部の入力運用が全社の未入力率を押し上げている主因。足利営業部（10%）が模範。

### 月別の偏り（直近）

| 月 | 未入力/総数 | 未入力率 |
|----|-----------|--------:|
| 2025-10 | 24/66 | 36% |
| 2025-11 | 15/33 | 45% |
| 2025-12 | 21/43 | 49% |
| 2026-01 | 15/26 | 58% |
| 2026-02 | 9/32 | 28% |
| 2026-03 | 5/19 | 26% |

→ 2025末〜2026年初に未入力率が上昇。慢性的というより運用の緩みも示唆。

### 対応方針

- **自動補完は行わない**（推測で理由を埋めると失注要因分析が汚染されるため）。`lost_reason_missing` フラグのみ付与。
- 営業部向け入力ガイドラインを Obsidian に作成し、入力を促進（特に埼玉営業部）。

```sql
SELECT id, sales_dept, score FROM past_cases WHERE quality_flag LIKE '%lost_reason_missing%';
```
