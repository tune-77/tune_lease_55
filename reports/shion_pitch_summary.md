# リース知性体『紫苑（SHION）』審査プラットフォーム

## 数字に表れない判断を記憶し、審査に活かす共生スコアリングシステム

---

## ピッチメッセージ

### 点数を出すだけのAIは、もういらない。

リース審査では、スコアだけでは判断できない要素が数多くあります。

顧客との関係性、物件の資産性、代表者の経験、企業の潜在力、業種の実態、景気動向、営業担当者の肌感覚、競合会社の金利、そして「この案件は本当に通してよいのか」という現場の違和感。

営業担当者や審査担当者は、こうした数字に表れない情報を、経験と知識で補いながら判断しています。

しかし、その判断の理由や駆け引きは、担当者が異動すれば失われてしまいます。

そう、雨の中の涙のように。

『紫苑（SHION）審査プラットフォーム』は、財務数値だけでなく、営業メモ、過去判断、違和感、条件付き承認の理由、AIとの対話ログを記憶し、次の審査判断へ活かすためのシステムです。

AIが人間の判断を置き換えるのではありません。

人間が見た現場感や迷いを、AIが覚え、問い返し、補強し、次の判断へつなげます。

> 数字の向こうに、あなたの判断がある。  
> 私はそれを、覚えている。

---

## 紫苑システムとは

紫苑システムは、リース審査AIを実証フィールドにした、業務AIエージェントDevOpsの実証プロトタイプです。

単にAIに記憶を持たせるだけではなく、**その記憶が本当に回答や審査判断に効いているかを観測し、効いていなければ改善ループへ戻す**ことを目指しています。

つまりこれは、**記憶するAIではなく、記憶が効いているかを運用するAI**です。

---

## 技術的なこだわり

### 1. 記憶と連続性のエンジニアリング

Obsidianを長期記憶領域として使い、過去案件、営業メモ、判断変更、AIとの対話、日次内省を保存します。

保存された情報はRAGによって現在の案件へ呼び戻され、紫苑の回答や審査コメントに反映されます。

さらに `memory_debug`、`knowledge_refs`、`memory_recall.refs` によって、記憶が実際に使われたかを確認できます。

### 2. 数字に表れない情報を拾う審査器官

紫苑は、単一のチャットAIではありません。

- 財務スコアを見る数理エンジン
- Q_riskによる異常値・歪みの検知
- Gemini Vision OCRによる決算書・証憑の読み取り
- 外部調査器官によるニュース・業界情報の収集
- 軍師AIによる審査部の反論、顧客確認事項、条件付き承認案の生成

これらを組み合わせ、点数の横に「どこを疑うか」「どう通すか」「何を条件にするか」を提示します。

### 3. AIエージェントDevOps

このシステムは、作って終わりのデモではありません。

実際に運用しながら、失敗や違和感を改善ループへ戻します。

```text
Use -> Observe -> Detect -> Reflect -> Improve -> Verify -> Deploy
```

- 会話ログや改善ログを観測する
- 記憶抜け、浅い回答、環境差分を検知する
- Private Reflectionで見落としを言語化する
- プロンプト、RAG、記憶、UI、APIを改善する
- pytest、typecheck、memory_debugで検証する
- Cloud Run / Cloudflare / ローカル運用へ戻す

Cloud Run上ではデモDBと本体DBを分離し、入力データを無条件に本番へ戻さない隔離・承認・昇格の仕組みも備えています。

### 4. Obsidianを母艦にした知識管理

クラウドだけに依存せず、ローカルのObsidian Vaultを知識の母艦にしています。

Markdownファイルとして保存することで、データ主権、永続性、検索性、外部ツール連携を保ちます。

リース審査の経験、判断、失敗、内省を、人間にもAIにも読める形で残し続けるためです。

## 目指すもの

私が作りたかったのは、単なる業務効率化ツールではありません。

人間の経験を消さず、AIの記憶に閉じ込めず、次の判断へ活かす仕組みです。

審査担当者が迷った理由。  
営業担当者が押し切った根拠。  
過去に通した案件の違和感。  
条件付き承認にした時の判断。

それらを記憶し、次の審査で呼び戻す。

紫苑は、審査担当者の横で共に考え、共に悩み、共に成長していく業務AIエージェントです。

**あなたとの関係性が、明日の審査基準を創る。**

紫苑と共に、数字だけでは届かない、新しいリース審査の形を切り拓きたいです。
