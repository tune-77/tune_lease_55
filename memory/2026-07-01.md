## Next standalone CSS欠落の再発防止
- Cloudflare版でCSSが死んだ原因は、Next standalone実行側 `frontend/.next/standalone/.next/static` に、HTMLが参照する `/_next/static/css/*.css` が同期されていなかったことだった。
- `run_next_stable.sh` に `sync_standalone_assets()` を追加し、ビルド実行時だけでなく、ビルドスキップ時・既存プロセス再利用時・`RESTART_SCOPE=next` のNext-only再起動前にも `frontend/.next/static` と `frontend/public` をstandalone側へ同期するようにした。
- Next-only再起動で `lsof -ti :3000` がCloudflare接続プロセスまで巻き込む可能性があったため、API/Nextの停止対象を `lsof -tiTCP:<port> -sTCP:LISTEN` に限定した。影響: Nextだけ再起動してもCloudflare tunnelを誤って落としにくくなる。
- 検証: `bash -n run_next_stable.sh` 成功。修正後の `RESTART_SCOPE=next` で `Synced standalone static/public assets.` が先に出ることを確認。Cloudflare新URL `https://stable-payment-executed-listprice.trycloudflare.com/home` とCSS assetがともに `200 OK`。
- その後、対話AIで一時的に Cloudflare 502/530 が出た。原因はNext再起動中にCloudflareが `127.0.0.1:3000` へ到達できなかった数秒間のエラー。最新URL `https://stable-payment-executed-listprice.trycloudflare.com` では `/lease-intelligence` が `200 OK`、`/api/lease-intelligence/dialogue` のPOSTもローカル/Cloudflareともにreply返却を確認。
- 追加対策として、`/lease-intelligence` の対話AI送信に `postDialogueWithRetry()` を追加し、Cloudflareの一時的な `502/503/530`、origin unreachable、connection refused、FastAPI proxy接続失敗だけを短時間で最大3回リトライするようにした。通常のAPI/Geminiエラーは従来通り表示する。影響: Next/Cloudflare再起動直後の数秒エラーをユーザーへ出しにくくなる。
- `run_next_stable.sh` の `RESTART_SCOPE=next` は、Nextを落とした後に `http://127.0.0.1:3000/` が戻るまで待ってから終了するようにした。検証: `frontend` の `npm run build` 成功、Next-only再起動後に Cloudflare `/lease-intelligence`、CSS asset、対話AI POST がすべて成功。
