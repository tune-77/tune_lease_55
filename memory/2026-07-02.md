# 2026-07-02

## Full審査APIのStreamlit依存切り離し
- Userが「スコアが変わってもいいからやろう」と判断したため、`/api/score/full` の既定経路を旧Streamlitエンジン `run_full_scoring_api()` から、Streamlit非依存の `scoring_core.run_full_api_scoring()` へ切り替えた。
- 旧Streamlit経路は `USE_LEGACY_STREAMLIT_FULL_SCORE=1` で戻せる退避経路として残した。影響: 審査結果が旧経路と完全一致しなくても、FastAPI本体がStreamlitモック・物理ブリッジ・子プロセスに依存せずにFull審査を返せる。
- `scoring_core.py` に `run_full_api_scoring()` を追加し、既存 `run_quick_scoring()` を互換実装として利用する形にした。名前上もFull APIの本流とQuick互換を分けた。
- 純API化後、デモ精密工業は旧Streamlit経路 `score=89.9` から純API経路 `score=81.9` に変わった。判定はどちらも `承認圏内`。Userが許容した「変わってもいい」の範囲。
- Cloudflare/Next proxyで30秒切断が出たため、同期Full審査からMahalanobis/UMAP診断を既定で外し、`ENABLE_SYNC_SCORING_DIAGNOSTICS=1` の時だけ同期実行するようにした。影響: Full審査の応答時間は単体で初回約0.77秒、2回目約0.01秒まで短縮。診断値は既定で `null` になる。
- 検証: `python -m py_compile api/main.py scoring_core.py api/scoring_full.py api/scoring_worker.py components/score_calculation.py` 成功。`frontend` の `npm run build` 成功。純API単体 `run_full_api_scoring()` は `score=81.9` / `hantei=承認圏内`。
- 注意: 最後のNext/FastAPI/Cloudflare再起動検証で、昇格実行がusage制限により使えず、通常サンドボックス権限で起動したNextが `listen EPERM 127.0.0.1:3000` で再起動ループになった。フォアグラウンド起動は停止済み。公開環境への再反映は、昇格実行が戻った後に `RESTART_SCOPE=api` と `RESTART_SCOPE=next` を再実行して確認する必要がある。
