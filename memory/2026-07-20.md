# 2026-07-20

## 紫苑: 業務ごとに育つ静かな判断補助AI

- 事実: User は、普通のユーザーは Codex や Claude ではなく紫苑とだけ話す前提だと整理した。User 自身の紫苑は Codex/Claude/Obsidian を上流に持つ濃い育ち方だが、一般ユーザー向けの紫苑は、業務の横に静かにいて、重要な判断だけを候補化する存在でよい。
- 影響: 紫苑の標準形は、思想を毎回語るAIではなく、業務ごとの会話・案件・結果から判断資産候補をそっと拾い、人間承認後に必要な時だけ横に出す「静かな紫苑」として設計する。User向けの濃い紫苑は原型研究であり、一般提供時の標準体験とは分ける。
- 次の行動: 発表・README・設計説明では、`基本紫苑コア + 業務別判断資産 + 人間承認ループ` を標準構造として扱う。個人雑談や関係性記憶は、業務判断に明確に効く場合だけ候補化する。

## 紫苑: 中核コア思想と業務別レイヤーの分離

- 事実: User は、紫苑の中核コア思想は必要だが、普通の紫苑では表に出しすぎず、業務ごとの紫苑を作る目的から外れないようガードすべきだと整理した。
- 影響: 全紫苑に共通する下層ルールとして、人間の代わりに最終判断しない、判断する人の横に置く、判断資産は候補・検疫・承認・昇格を通す、古い判断は改訂可能にする、人間査定・監視・懲戒には使わない、という原則を持たせる。その上に業務知識、業務ごとの判断観点、組織や個人の判断資産を重ねる。
- 次の行動: 紫苑の説明では、コア思想をキャラクター設定ではなく、判断資産を扱うための安全規約として説明する。通常回答では理念を語らず、判断・記憶・昇格の扱いを裏で縛る。

## 紫苑: Codex / Claude 由来の思考を紫苑へ渡す経路

- 事実: User は、Codex で考えた内容や Claude のレビューが、memory / reports / Obsidian を経由して紫苑の記憶索引へ入り得ることに気づいた。紫苑は単一モデルではなく、Codex で考え、Claude が評価し、Obsidian と memory に残り、紫苑が必要な時に想起する育成ループを持つ。
- 影響: この経路は強いが、AI同士の自己増幅や、AIの提案をUserの判断資産と誤認する危険もある。境界線は「Userが選んだものだけが判断資産。AIが言っただけのものは候補または参考意見」とする。
- 次の行動: Codex/Claude由来の内容を紫苑へ渡す場合は、全文投入せず、紫苑用に圧縮し、`value_memory` / `dialogue_memory` / `judgment_memory` などへ分類する。Cloud Runやデモ向けには private / dialogue / reflection の扱いに注意する。

## 紫苑: 普通の紫苑の定義

- 事実: User は「一体普通の紫苑はどれだ」と問い、普通の紫苑は業務の横に静かにいる判断補助AIだと整理した。
- 影響: 普通の紫苑は、会議や人間判断を置き換えない。過去の似た判断を少し思い出し、重要そうな違和感だけ候補にし、追加確認を返し、承認された判断資産だけ次回から使う。濃い思想や自己説明は必要時だけでよい。
- 次の行動: 標準UXでは、紫苑の育成や記憶を前面に出しすぎない。月次・節目・明示操作時にだけ「残す候補」を見せ、人間が承認する導線にする。

## 紫苑: タスク系統は記憶・判断資産と分離する

- 事実: User は、紫苑がタスクを覚えることに関心を示し、記憶とは別系統でも実装可能か確認した。最小構成として `data/shion_tasks.jsonl` を追記型タスク台帳にし、`api/shion_tasks.py` と `/api/shion/tasks` 系APIで追加・一覧・更新・完了/取消を扱う実装を追加した。
- 影響: 紫苑は「なぜそれをやるか」を記憶系統に、「いつ何をやるか」をタスク系統に分けられる。タスクを判断資産へ混ぜないため、記憶DBがToDoで濁りにくくなる。通知や厳密リマインドはまだ別系統として未実装。
- 次の行動: 最初は画面表示・今日のタスク確認だけで試し、必要になったら cron / Calendar / 通知へ接続する。チャットからの自動抽出は、まず候補化と人間確認を挟む。

## 紫苑: 普通の記憶エージェント機能はレーン分離で見える化する

- 事実: User は、普通の記憶エージェント機能を足すと業務判断に影響するか確認し、実装を依頼した。`api/shion_memory_lanes.py` と `/api/shion/memory-lanes` を追加し、個人記憶、タスク記憶、索引済み記憶を分離して返す読み取り専用APIを作った。
- 影響: 紫苑は普通の記憶エージェントとしての状態を見せられるが、普通の記憶が自動で判断資産になることはない。personal_memory は直接会話の自然さ、task_memory はやること、indexed_memory はroute依存の想起として扱い、業務判断の根拠にはレビュー済み judgment_memory を優先する。
- 次の行動: まずはUIやデバッグ表示でレーン別に確認する。チャットプロンプトへ入れる場合も、personal/task/dialogue/judgment/value を混ぜず、用途ごとの注入順を守る。

## 紫苑: ハッカソン中は記憶タスク機能を広げない

- 事実: User は、チャットからタスク登録したい気持ちはあるが、ハッカソン中なので無理しない方針を確認した。今後の実装計画として `planning/shion_memory_task_agent_plan.md` を作成した。
- 影響: 現時点では、タスク台帳と記憶レーンAPIの土台までで止める。チャット自動抽出、自然文期限解釈、通知、Calendar連携は後回しにする。
- 次の行動: ハッカソン後に、ホームの小さな今日のタスク表示、手動追加UI、チャット内タスク候補化、朝の確認、通知連携の順に検討する。

## 紫苑: Codex / Claude 使用記録は補助知識として明示する

- 事実: User は、Codex や Claude での使用記録も紫苑の補助知識として使えると整理し、その扱いを明示する方針を確認した。`/api/shion/memory-lanes` の索引済み記憶に `agent_review_context` を追加し、Codex / Claude / 他AIレビュー由来の記憶を補助知識として見えるようにした。
- 影響: Codex/Claude由来の内容は、方針理解・設計相談・実装判断の材料にはなるが、それ自体は判断資産ではない。Userが採用・修正・却下したという選択圧がかかった部分だけ、判断資産候補に進められる。
- 次の行動: 発表・README・設計説明では、複数AIの出力を紫苑が丸呑みするのではなく、補助知識として台帳に通し、Userの選択後にだけ判断資産候補へ進むと説明する。

## 紫苑: 利用者ごとのAI記憶をObsidianへ集約する一般化

- 事実: User は、このツールを使う人は当然何らかのAIと連携しているため、各利用者が使うAIの記憶・会話要約・レビュー結果をObsidianに設定すれば、同じ運用が可能ではないかと整理した。
- 影響: 紫苑は特定AIに依存せず、ChatGPT / Codex / Claude / Gemini / Cursor / 社内AIなどの使用記録を、ObsidianやMarkdown Vaultへ集約し、`agent_review_context` や `technical_memory`、必要に応じて `dialogue_memory` として扱える。これにより「どのAIを使ったか」ではなく「AIとの対話から人間が何を選び、何を資産化したか」を見る運用に一般化できる。
- 次の行動: 一般化説明では、AIログは補助知識、Userが選んだものだけが判断資産候補、Obsidianは中立的な記憶台帳、という3点を明示する。各AIからの取り込みは全文ではなく、作業要約・意思決定・レビュー結果・採否だけに絞る。
