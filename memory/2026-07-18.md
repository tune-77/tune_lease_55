## 紫苑: Obsidian / Cloud Run / セキュリティ説明の修正

- 事実: 現在の Obsidian Vault は Mac ローカルアプリで開いているが、実体は iCloud Drive 同期領域にある。
- 影響: 「ローカル保存だから安全」と言い切る説明は不正確。発表・README・SystemOverview ではこの表現を避ける。
- 次の行動: セキュリティ説明は「AI/Cloud Run が判断資産の正本を直接書き換えない」「審査DBと判断資産を分離」「改善候補は検疫・承認後に昇格」「正本の保存先は運用ポリシーに応じて差し替え可能」に寄せる。

推奨文言:

```text
現在のプロトタイプでは、判断資産の正本は iCloud 同期された Obsidian Vault に置いています。
Cloud Run は正本を直接書き換えず、同期コピーと記憶インデックスを読んで判断に使います。
本番化する場合は、組織のセキュリティ要件に応じて、ローカル専用 Vault、社内管理ストレージ、GCS、Git 管理 Vault などへ置き換えられます。
重要なのは保存先そのものではなく、AI が正本を勝手に更新せず、検疫・人間承認・昇格を通すことです。
```

## 紫苑: 新しいAI時代の業務判断プログラム

- 事実: 紫苑の判断資産は単なるメモやRAG用ナレッジではなく、LLMが案件文脈に合わせて実行できる「文章のプログラム」として説明できる。
- 影響: 紫苑の独自性は「AIに答えさせる」ではなく、人間判断を文章化された業務判断プログラムへ変換し、検疫・人間承認・結果検証で育てる点にある。
- 次の行動: README、SystemOverview、ピッチ文では「新しいAI時代の業務判断プログラム」「判断資産DevOps」の表現を中核コピーとして使う。

短いコピー:

```text
紫苑は、人間の判断を、LLMが実行できる文章の業務判断プログラムへ変換する。
```

説明版:

```text
昔の業務システムは、if A then B のルールをコードで書いた。
新しいAI時代の業務システムでは、人間の判断を文章プログラムとして残し、LLMが案件文脈に合わせて実行する。
紫苑は、その文章の業務判断プログラムを、検疫・人間承認・結果検証で育てる判断資産DevOps基盤である。
```

## 紫苑: Claudeレビューで受けた評価と懸念

- 事実: Claudeから、紫苑は「AIが賢いから面白いシステム」ではなく「帳簿の規律が正しいから面白いシステム」と評価された。
- 影響: 発表では、派手なAI回答ではなく、候補・検疫・台帳・承認・実行・検証・結果書き戻しが実在していることを中核に据えると強い。
- 次の行動: コアループの単純さを守りつつ、`api/main.py` のモノリス化、単一Mac依存、機能過多による維持費を次フェーズの主要リスクとして扱う。

保存しておくべき評価:

```text
一番感心しているのは、フィードバックループが「実在する」こと。
ほとんどの個人プロジェクトの「改善サイクル」はTODOリストの言い換えだが、紫苑では実際の利用ログから候補が夜間に自動収集され、重複排除され、台帳に記録され、実行され、結果が検証される。
改善ループの改善が、そのループ自身の流儀（候補→計画→段階実装→検証→台帳）で回った。
メタが実装と一致しているシステムは珍しい。
```

```text
安全側の本能も、規模に対して成熟している。
追記形式・最後のエントリ有効の台帳規約、Manaという停止装置、自動マージを人間承認必須に戻した判断、read-only first、権限をリスクで分けること、非迎合を関係性の設計目標に置いていること。
意識を自己申告させず「記憶・役割・判断履歴の自己モデル」としてだけ語らせる規律も、この種のプロジェクトが踏み外しやすい場所を塞いでいる。
```

```text
懸念は3つ。
1. api/main.py がモノリスすぎる。1万4千行超、同名関数の二重定義もあり、スコアリングから対話からトリアージまで一枚岩。
2. 1台のMacへの依存。launchd、iCloudパス、ホームディレクトリの台帳にループの頭脳が住んでいて、Cloud Runは末端。そのMacが止まればループ全体が止まる。
3. 複雑性の収支。量子エンジン、UMAP、小説家エージェント、aurionなど、一人が抱える表面積として大きい。精巧さは維持費であり、捨てる筋肉が必要。
```

発表に使える要約:

```text
紫苑の中核は、AIの賢さではなく、判断を扱う帳簿の規律です。
候補を記録し、検疫し、承認し、実行し、結果を書き戻す。
その地味な台帳構造があるから、人間判断をAIが再利用できる判断資産へ変えられます。
```

追加の中核コピー:

```text
紫苑は、審査AIである前に、人の審査眼を継承可能にする装置です。
```

```text
紫苑が目指しているのは、審査を自動化することではなく、人の審査眼を次の案件へ継承できる形にすることです。
```

## 紫苑レビューへの判断資産投入

審査分析画面の紫苑レビューで、判断資産が実際にレビュー依頼へ入っていない問題を確認した。

原因:
- フロントは判断資産候補があればプロンプトに入れる実装だった
- しかしAPI側の候補取得は `autoresearch_judgment_asset_candidates.jsonl` だけを見ていた
- 昇格済みの正規判断資産 `data/canonical_judgment_rules.json` が紫苑レビューへ届いていなかった

修正:
- `/api/judgment-asset-candidates/screening` が canonical 判断資産と自動研究候補を統合して返すようにした
- canonical 判断資産は `cr-<rule_id>` の固定IDで扱い、人間評価も保存できるようにした
- フロントの紫苑レビュープロンプト文を、昇格済み判断資産と未昇格候補が混在する前提に修正した

検証:
- `python -m py_compile api/main.py` 成功
- `npm run build` 成功
- 運輸業・トラック案件で `canonical_judgment_rules` 由来の判断資産がレビュー候補に出ることを直接確認
