#### P1. Gemini API を有料プランへ切り替え

**問題:**
無料枠の上限（15リクエスト/分）を複数人の同時操作で簡単に超過する。エージェント議論1回で10〜15リクエスト発生するため、2人が同時に操作すると即 429 エラーになる。

**対策:**
Google AI Studio または Google Cloud で有料プランに切り替える。

**費用:**
gemini-2.0-flash の場合、3〜5人 × 1日10操作 = 50〜200回/日
→ 月額数百〜1,000円程度（利用量次第）

**作業内容:**
1. Google Cloud プロジェクトで請求を有効化
2. 新しい API キーを発行
3. `.streamlit/secrets.toml` の `GEMINI_API_KEY` を更新

**担当:** —
**工数:** 1時間
**依存:** なし

---

#### P2. Ollama 同時リクエスト対応

**問題:**
Ollama はデフォルトで同時1リクエスト処理。複数人が LLM 呼び出しを行うとキュー待ちが発生し、最大で人数 × 処理時間（30〜120秒）の待機が生じる。

**対策 A（推奨）:** Gemini 統一
AI エンジンをデフォルト Gemini に変更し、Ollama はオプション扱いにする。

**対策 B:** Ollama 並列処理数の増加
GPU の VRAM が十分ある場合、環境変数で並列数を増やす。
```bash
OLLAMA_NUM_PARALLEL=3
```

**作業内容:**
- 対策 A: `config.py` の `OLLAMA_MODEL` を fallback 扱いに変更、UI のデフォルト選択を Gemini に修正
- 対策 B: サーバー起動スクリプトに環境変数を追加

**担当:** —
**工数:** 対策 A = 2時間、対策 B = 30分
**依存:** P1（Gemini 有料化）が先行推奨

#### P3. `st.cache_data.clear()` の影響範囲を限定

**問題:**
設定画面・分析画面で `st.cache_data.clear()` が呼ばれており、これは**全ユーザーの全キャッシュを一括クリア**する。ユーザーAの操作でユーザーBのキャッシュが消え、重い再読み込みが発生する。

**対象箇所（要確認）:**
- `components/settings.py`
- `components/analysis_results.py`
- その他 `st.cache_data.clear()` を呼び出している箇所

**対策:**
`st.cache_data.clear()` を削除し、TTL によるキャッシュ自動失効に委ねる。または特定の関数キャッシュのみを `.clear()` する記法に変更する。
```python
# 変更前
st.cache_data.clear()

# 変更後（特定関数のキャッシュのみクリア）
load_benchmarks.clear()
load_jsic.clear()
```

**担当:** —
**工数:** 半日
**依存:** なし

#### P4. JSONL ファイルの SQLite 移行

**問題:**
`case_news.jsonl`（現在 103KB・増加中）, `agent_team_logs.jsonl` 等はファイル追記方式のため、同時書き込みで行破損のリスクがある。また全件読み込み時のメモリ消費がデータ増加とともに増える。

**対象ファイル:**
| ファイル | 現在のサイズ | 移行先テーブル案 |
|---|---|---|
| `data/case_news.jsonl` | 103KB | `case_news` テーブル |
| `data/agent_team_logs.jsonl` | 23KB | `agent_team_logs` テーブル |
| `data/debate_logs.jsonl` | 6KB | `debate_logs` テーブル |
| `data/coeff_history.jsonl` | 1KB未満 | `coeff_history` テーブル |

**対策:**
既存の `data/lease_data.db` に上記テーブルを追加し、追記処理を SQLite INSERT に変更。WAL モードを有効化することで同時書き込み耐性を向上。

```python
# WAL モード有効化（接続時に1行追加）
conn = sqlite3.connect(DB_PATH)
conn.execute("PRAGMA journal_mode=WAL")
```

**担当:** —
**工数:** 1〜2日
**依存:** なし（WAL モードだけなら30分）

#### P5. 係数設定の管理者限定化

**問題:**
`coeff_overrides.json` の変更（スコア係数の手動調整）を全ユーザーが行える。誤操作や意図しない変更でスコアリング結果が変わるリスクがある。

**対策:**
`components/settings.py` の係数変更 UI を `session_state["user_role"] == "admin"` の場合のみ表示する。

```python
# components/settings.py に追加
if st.session_state.get("user_role") != "admin":
    st.info("係数設定の変更は管理者のみ行えます。")
    return
```

**担当:** —
**工数:** 2時間
**依存:** `auth_logic.py` の `user_role` 対応（対応済み）

#### P6. SQLite → PostgreSQL 移行（10人超の場合）

**問題:**
SQLite は同時書き込みに構造的な限界がある（WAL モードでも書き込みは逐次処理）。10人を超える規模では応答遅延が顕著になる可能性がある。

**対策:**
PostgreSQL（ローカルまたは Supabase 等クラウド）への移行。
`sqlite3` → `psycopg2`（または `SQLAlchemy`）に変更。

**費用:**
Supabase 無料枠: 500MB DB、2プロジェクト
本番利用: $25/月〜

**担当:** —
**工数:** 3〜5日
**依存:** P4（JSONL の SQLite 移行）が完了していること

#### P8. Gemini SDK の移行（google-generativeai → google-genai）

**問題:**
`ai_chat.py:130` で使用している `google.generativeai`（`import google.generativeai as genai`）は
2025年に **deprecated（廃止）** となった旧SDK。現時点では動作するが、将来のアップデートで
予告なく動かなくなるリスクがある。

```python
# 現状（廃止済み）
import google.generativeai as genai
genai.configure(api_key=api_key)
genai.GenerativeModel(model)

# 移行先（新SDK）
import google.genai as genai
client = genai.Client(api_key=api_key)
client.models.generate_content(model=model, contents=prompt)
```

**参考:** [google-gemini/deprecated-generative-ai-python](https://github.com/google-gemini/deprecated-generative-ai-python)（★2,273・アーカイブ済み）

**対象箇所:**
- `ai_chat.py` の `_gemini_chat()` 関数（line 116〜170）
- `ai_chat.py` の `_stream_gemini()` 関数（line 794〜）

**作業内容:**
1. `pip install google-genai` を追加（`requirements.txt` 更新）
2. `ai_chat.py` の `_gemini_chat()` を新SDKの呼び出し形式に書き換え
3. `_stream_gemini()` のストリーミング処理を新SDK形式に書き換え
4. 動作確認（429エラーハンドリング・空レスポンス処理が引き継がれることを確認）

**担当:** —
**工数:** 半日〜1日
**依存:** なし（多人数対応 P1〜P7 とは独立）

#### P7. LangGraph SqliteSaver によるエージェント議論の永続化

**問題:**
現在エージェント議論の進行状態は `st.session_state` のみに保存されており、ページリロードで消える。

**対策:**

**工数:** 3〜5日（`agent_team.py` のリアーキテクチャを含む）
**依存:** P4 が完了していること。LangGraph の学習コストあり。
