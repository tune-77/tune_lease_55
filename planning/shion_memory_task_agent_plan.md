# 紫苑: 記憶エージェント / タスク系統 実装計画

作成日: 2026-07-20

## 方針

ハッカソン中はこれ以上広げない。現在は API と台帳の土台までで止める。

紫苑の記憶エージェント機能は、業務判断を濁らせないためにレーン分離を守る。

- 個人記憶: 直接会話の自然さに使う
- タスク記憶: いつ何をやるかを管理する
- 対話記憶: 継続文脈・好みを見る
- 補助知識: Codex / Claude / 他AIレビューの使用記録。方針理解や設計相談の材料にはするが、Userが選んだもの以外は判断資産にしない
- 判断記憶: 業務判断に再利用する
- 価値記憶: 人間補助・安全境界を守る
- 技術記憶: 実装・運用制約に使う

## 現在できていること

- `data/shion_tasks.jsonl`
  - 追記型タスク台帳
  - 追加、一覧、更新、完了/取消
- `/api/shion/tasks`
  - タスク管理API
- `/api/shion/memory-lanes`
  - 個人記憶、タスク、索引済み記憶を分離して見える化
  - Codex / Claude 由来の `agent_review_context` は補助知識として明示し、判断資産とは分ける
- テスト
  - `tests/test_shion_tasks.py`
  - `tests/test_shion_memory_lanes.py`

## 実装順

### Phase 1: 画面に小さく出す

目的: 存在を見えるようにするだけ。チャット挙動には干渉しない。

- ホームに「今日のタスク」3件を表示
- 完了ボタンを置く
- 記憶レーンの件数だけ表示
- 本文や個人記憶の詳細は出しすぎない

### Phase 2: 手動追加UI

目的: チャット自動抽出の前に、人間が明示的に登録できる導線を作る。

- タスク名
- 期限
- メモ
- タグ
- 登録ボタン

期限は最初は ISO 日付/日時のみ。自然文解釈はまだやらない。

### Phase 3: チャット内タスク候補化

目的: 自動登録ではなく、候補化して人間確認を挟む。

候補にする表現:

- `〜をタスクにして`
- `あとで〜する`
- `明日〜する`
- `決勝前に〜する`

紫苑の返し方:

```text
タスク候補にします。

- タスク: ...
- 期限: ...
- 種別: task

登録しますか？
```

Userが `登録して` と言った場合だけ `/api/shion/tasks` に登録する。

### Phase 4: 起動時 / 朝の確認

目的: 通知ではなく、画面表示で軽く思い出せるようにする。

- ホーム greeting の下に今日のタスクを表示
- 期限切れタスクを上に出す
- 完了/延期だけできるようにする

### Phase 5: 通知連携

目的: 本当に必要になってから実装する。

候補:

- cron
- Calendar
- OS通知
- Cloud Run / GCS writeback

ここはハッカソン後に検討する。

### Phase 6: 利用者ごとのAIログ取り込み

目的: 紫苑を特定AIに依存させず、利用者が普段使っているAIとの会話・レビュー・作業記録を補助知識として使えるようにする。

対象:

- ChatGPT / Codex
- Claude
- Gemini
- Cursor
- Devin
- 社内AI

取り込み先:

- Obsidian / Markdown Vault
- Daily
- AI Logs
- Reviews
- Decisions
- Memory Candidates

扱い:

- AIログは `agent_review_context` として扱う
- AIログは判断資産そのものではない
- Userが採用・修正・却下した部分だけ判断資産候補に進める
- 全文ではなく、作業要約・意思決定・レビュー結果・採否だけを保存する
- ObsidianはAI横断の中立的な記憶台帳として扱う

## 禁止事項

- チャットから勝手にタスク登録しない
- `覚えて` をすぐタスク扱いしない
- タスクを判断資産に混ぜない
- Codex / Claude が言っただけの内容を判断資産にしない
- 他AIログを全文投入しない
- 個人記憶を業務判断の根拠にしない
- ハッカソン中に通知系まで広げない

## 判断基準

次に進める条件:

- APIが安定している
- ホーム表示が邪魔にならない
- タスク台帳がノイズで増えない
- Userが手動登録を使いたいと感じる

止める条件:

- チャットが重くなる
- タスク候補の誤検出が多い
- 判断資産とタスクが混ざる
- デモ前の安定性を損なう
