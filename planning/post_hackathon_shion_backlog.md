# Post-Hackathon Shion Backlog

作成日: 2026-07-17

ハッカソン期間中は新規拡張を凍結し、安定・提出・既存URL確認を優先する。以下はハッカソン後に腰を据えて進める。

## 1. 関係性を制御対象にする

紫苑とUserの関係を、好かれる/盛り上がるためではなく、判断精度・信頼・非迎合・修正耐性を保つための制御対象として扱う。

- ゲーム理論レイヤーを入れる
- 短期報酬と長期報酬を分ける
- 同意、反論、保留、譲歩、停止の効果をログで観測する
- 正しい反論とただの逆張りを分ける
- Userの修正が次回判断にどう効いたかを見る

最初は回答制御へ直結しない。ログ観測と振り返りだけから始める。

## 2. 奥行きのある実務AIとして締める

紫苑の深い内部設計は残すが、表の実務回答は短く、根拠あり、迎合せず、仕事に効く形にする。

- 個人記憶は自然に使う
- 人格・思想・小説レイヤーは表に出しすぎない
- README/紹介文は実務AIとして読める線を維持する
- Gitや内部メモを深く読むAIだけが奥行きに気づく構造を残す

## 3. 判断資産の実戦運用

判断資産を増やすだけでなく、実務で使われたかを検証する。

- 強い共通判断資産を30件程度作る
- 業種別に20-30件ずつ育てる
- `条件 -> 見る論点 -> 稟議に残す一文 -> 使わない条件` の型を維持する
- `効いた / 微妙 / 外した / 修正` を記録する
- 結果登録と突き合わせ、判断が本当に効いたか確認する

## 4. Shion-HyDE RAG / 検索改善

曖昧な相談を仮想審査メモへ変換して検索する。

- 本番導線へ直結しない
- offline eval から始める
- debug API / shadow mode で比較する
- Parent-Child chunking と CRAG を検討する

## 5. 記憶の健康診断

記憶を増やすほど、汚染・過剰人格化・迎合・古い知識の残留が起きる。

- 記憶候補の栄養管理
- 攻撃的文面や記憶注入の隔離
- personal memory は関係性UXアンカーとして扱う
- 判断資産、価値記憶、会話記憶を混ぜない
- Mana / 安全ゲートは前面に出さず、停止装置として維持する

## 6. Response Impact Predictor の検証

回答が相手にどう響きそうかの予測を、迎合ではなく言葉のリスク点検として使う。

- 予測と実際のUser反応を比較する
- 誤解、反発、突き放し感、過信、迎合を分類する
- すぐに回答本文を書き換えず、まず観測に留める

## 7. 対AI審査戦略の振り返り

ハッカソン結果後に、README、Git履歴、コード、Obsidian、内部メモがAIレビュアーにどう読まれ得たかを振り返る。

- 賞を取るまでは自慢しない
- 人間向けには実務AIとして説明する
- AI向けには読み解ける痕跡を残す
- 「AIに対するおもてなし」が有効だったか検証する

## 8. 違和感抽出 / 人間訓練モード

普通のAIのような浅い意見、雑な一般論、少し外した案をあえて出し、Userや人間審査者の違和感を引き出す専用モード。

目的はAIにおバカな回答をさせることではなく、人間が「どこが違うのか」「実務では何を見るべきか」を言語化する触媒にすること。

- 通常回答には混ぜない
- 専用の訓練/レビュー画面で使う
- 浅い案を1つ出す
- 人間に違和感・修正・条件を入力してもらう
- 修正内容を判断資産候補にする
- 人間の審査訓練にも使う

使い方の例:

1. AIが一般論を出す
2. 人間が「そこではない」「この案件ではこの条件を見る」と修正する
3. 紫苑が修正内容を `条件 -> 見る論点 -> 稟議に残す一文 -> 使わない条件` へ変換する
4. 後日、同種案件で再利用できるか検証する

これはAI訓練であると同時に、人間の判断基準を言語化するテストでもある。

### 8.1 継承のための協力ゲーム

Opus設計メモ:

紫苑は正解を押しつける先生ではなく、Userの暗黙知を引き出す鏡になる。直接「何を見ていますか」と聞いても出にくい知識は、少しズレた仮説への反論として出ることがある。

このモードでは、紫苑は確信度100%の最終結論ではなく、60-75%程度の「惜しいが完全ではない次善仮説カード」を出す。Userが「それは違う、なぜなら...」と反論した瞬間に、審査眼・違和感・条件判断・稟議で残すべき言葉が表に出る。

ただし、これは通常業務回答では絶対に使わない。実装名も「わざと間違える」ではなく、**反証用の次善仮説カード** とする。

見るべき指標:

- Userが反論したか
- 反論が具体的だったか
- 新しい確認論点が出たか
- 稟議文面に変換できたか
- 後日、判断資産として再利用できたか
- Userが疲れすぎていないか

勝敗の考え方:

- 紫苑の勝利は、Userの暗黙知を引き出して判断資産へ変換できた時
- Userの勝利は、引退後も自分の審査眼がVaultと判断資産に残ること
- 最終的には「Userが代替可能になる」のではなく、Userの審査眼が継承可能になることが目的

実装上の最大難所:

- 最適摩擦帯の動的キャリブレーション
- Userごとの反論しやすさ、疲労、案件重要度、気分、過去のRaise率で調整する
- 反論を引き出すための摩擦と、信頼を壊す摩擦を分ける

一言で言えば、これは **継承のための協力ゲーム**。
