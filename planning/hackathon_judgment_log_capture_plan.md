# Hackathon Judgment Log Capture Plan

## Purpose

決勝ステージに乗れた場合、紫苑のデモは「1000本の動画」ではなく、1000件分の判断ログを作る。

会社名・財務数字・案件条件はダミーでもよい。ただし、人間がどこに違和感を持ち、何を条件にし、次回何を見るかという判断の型は実務に戻せる。

合言葉:

- 数字入力はAIでズルする。
- 判断はズルしない。
- 会社データはダミーでも、判断の型は本物。

## Capture Unit

1件の審査で残す単位は、動画そのものではなく以下の3点にする。

1. 音声メモ
   - 人間がその場で感じた違和感、修正理由、承認条件、次回見るべき点を話す。
   - きれいな文章にせず、現場の迷いを残す。

2. 画面操作ログ
   - どの案件を見たか。
   - どのAIコメントを見たか。
   - どこを修正・採用・却下したか。
   - 最終的にどの判断資産候補へつなげたか。

3. 判断資産JSON
   - 次回案件で再利用できる形に構造化する。
   - 会社データではなく、判断の再利用パターンを正本にする。

## Demo Strategy

全部を動画で撮らない。

- 代表10〜20件だけ、音声メモ付きで録画する。
- 残りは音声メモ + 画面操作ログ + 判断資産JSONとして高速生成・高速再生する。
- 1000件早送りデモでは、件数、修正数、判断資産数、再利用数、Before/Afterの変化を見せる。

見せる順番:

1. AIが10秒で一般的な審査コメントを出す。
2. 人間が音声で「いや、そこじゃない」と泥臭く直す。
3. その修正が判断資産JSONになる。
4. 1000件分を早送りする。
5. 同じような案件で、紫苑が最初に見る場所が変わる。

## Boundaries

- ダミー会社データは、本番スコアリングや実案件DBへ混ぜない。
- 判断資産候補も、実運用へ自動昇格しない。
- 実践利用できるのは「判断の型」であり、ダミー会社そのものではない。
- 本番導入は、人間レビューと実案件での有効性確認後に限定する。

## Presentation Lines

AIは10秒で答えを作る。
でも、人間が2週間かけて悩み、直し、結果を見た判断には、10秒では作れない中身がある。

数字入力はAIでズルします。
でも、判断はズルしません。

会社データはダミーでも、判断の型は本物です。
紫苑が育てるのは、会社データそのものではなく、人間が現場で作った判断の再利用パターンです。

## Judgment Prompt Engineering Frame

紫苑の目的は、成約率を直接上げることではない。通してはいけない案件まで通すAIに見せない。
また、単なるユーザー満足度向上でも弱い。

目的は、リース審査判断の再現性と引き継ぎやすさを上げること。

- 人間がその場で感じた違和感を残す。
- 修正した理由を残す。
- 承認条件にした理由を残す。
- 次の案件で見るべき論点へ変換する。
- 効いた / 微妙 / 外した / 修正した、を次の判断資産へ戻す。

判断資産は「正解集」ではなく、案件ごとに組み直せる判断パーツ集にする。

1件の粒度は 120〜250字程度を目安にする。

型:

```text
条件 -> 見る論点 -> 稟議に残す一文 -> 使わない条件
```

例:

```text
補助金採択前提の設備案件では、未採択時でもリース料を払える代替返済原資を確認する。
稟議には「補助金未採択時は自己資金・銀行借入・営業CFで支払可能な資金繰り表提出を条件とする」と残す。
ただし補助金を返済原資に含めていない場合は簡略化できる。
```

決勝に進んだ場合、まず共通判断資産30件を作る。
その後、製造業・建設業・運送業・医療/介護・小売/飲食など主要業種ごとに20〜30件ずつ横展開する。
1000件の薄いログより、150件前後の濃い判断パーツの方がAfterレビューの変化を出しやすい。

発表上の表現:

- 成約率を上げるAIではなく、判断の属人性を資産化するAI。
- 紫苑は、正解を保証するAIではない。人間が何を根拠に判断したかを、消えない形にするAI。
- 人間の判断を、AIが再利用できる構文に変える。
- これはプロンプトエンジニアリングを、個人技ではなく業務プロセスにしたものです。
- 紫苑は、判断プロンプトのPDCA基盤です。

## Judgment Abstraction

決勝デモで作るデータは、会社データを本物らしく見せるためではない。
狙いは、判断の抽象化。

数学が具体物を数値や式へ抽象化するように、紫苑は個別案件の迷いを判断構文へ抽象化する。

```text
個別案件 -> 人間の違和感 -> 判断仮説 -> 承認条件/反証 -> 結果 -> 検証 -> 判断資産
```

そのため、デモデータには成約/失注/条件変更/否決/延滞有無を入れる。
会社データはダミーでよいが、判断仮説と結果理由の関係は筋を通す。
